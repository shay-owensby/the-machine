# Mailchimp — API Reference (email-marketing plugin)
> Mailchimp Marketing API v3.0 — mainstream email marketing platform. Base URL: `https://<dc>.api.mailchimp.com/3.0` (datacenter-specific). Docs: https://mailchimp.com/developer/marketing/api/

## Capabilities summary

| Capability | Supported | How |
|---|---|---|
| Draft campaigns with custom HTML | Yes | `POST /campaigns` (type `regular`) then `PUT /campaigns/{id}/content` with `html` |
| Preview text | Yes | `settings.preview_text` on campaign create/update |
| A/B subject testing | Yes | `type: "variate"` campaign with `variate_settings.subject_lines` |
| Media/image library upload | Yes | File Manager: `POST /file-manager/files` (base64) |
| Reports API (read-only) | Yes | `GET /reports`, `GET /reports/{campaign_id}` |

## Authentication

- **Scheme**: HTTP Basic auth (`--user "anystring:$MAILCHIMP_API_KEY"` — username can be any string) or `Authorization: Bearer <key>`. Both work with API keys.
- **Create a key**: Mailchimp UI → profile icon → **Account & billing → Extras → API keys → Create A Key**. Keys inherit the full permissions of the user; there are no per-key scopes.
- **Env var convention**: `MAILCHIMP_API_KEY` in the client project's gitignored `.env`. Config files store only the env var *name*, never the value.
- **Datacenter quirk**: the key ends in a datacenter suffix, e.g. `abc123...-us21`. That suffix (`us21`) is the subdomain of the base URL: `https://us21.api.mailchimp.com/3.0`. Derive it in shell: `DC="${MAILCHIMP_API_KEY##*-}"`.
- **Rate/timeout quirks**: max 10 simultaneous connections; 120-second request timeout.

### Read-only connection test (GET /ping)

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
curl -sS --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/ping"
```

Success returns `{"health_status": "Everything's Chimpy!"}`. A health check that returns no account-specific data. (`GET /` returns account details + resource links if you need the account name for the wizard.)

## Fetching audiences/lists (setup wizard)

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
curl -sS --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/lists?count=100&fields=lists.id,lists.name,lists.stats.member_count"
```

Show the user: `lists[].id`, `lists[].name`, `lists[].stats.member_count`. The chosen `id` becomes `recipients.list_id` for campaigns. Also useful for defaults: `lists[].campaign_defaults.from_name`, `.from_email`, `.subject` (fetch without the `fields` filter to see them).

## Creating a draft campaign (core flow)

Mailchimp campaigns are created in `save` (draft) status and stay there until an **action** endpoint is called. We never call action endpoints (see FORBIDDEN below), so everything below produces drafts only.

**Step 1 — create the campaign shell** (`POST /campaigns`):

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
curl -sS -X POST --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/campaigns" \
  -H "Content-Type: application/json" \
  --data @campaign.json
```

`campaign.json` (minimal valid payload):

```json
{
  "type": "regular",
  "recipients": { "list_id": "abc123def4" },
  "settings": {
    "title": "2026-07 July Promo (internal name)",
    "subject_line": "Your July offer is here",
    "preview_text": "Save 20% through Sunday",
    "from_name": "Acme Co",
    "reply_to": "hello@acme.com"
  }
}
```

Response includes `id` (use for all subsequent calls), `web_id` (used in UI URLs), `status` (`"save"` = draft), `archive_url`.

**Step 2 — set the HTML content** (`PUT /campaigns/{campaign_id}/content`):

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
jq -Rs '{html: .}' < email.html > content.json   # safe JSON-escaping of the HTML
curl -sS -X PUT --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/campaigns/$CAMPAIGN_ID/content" \
  -H "Content-Type: application/json" \
  --data @content.json
```

**What the `html` field means — two paths:**

- **Path A — raw HTML (recommended when the email design is fully generated by this plugin):** `content.html` is the *entire rendered email*. Send a complete HTML document (`<!DOCTYPE html>...<html>...</html>`); Mailchimp does not wrap it in any platform-side template, header, or footer. ⚠️ unverified nuance — Mailchimp appends a compliance footer (unsubscribe link, postal address) at send time if the HTML lacks the required `*|UNSUB|*` / footer merge tags; confirm at https://mailchimp.com/developer/marketing/docs/campaigns/ . Best practice: include `*|UNSUB|*` and `*|LIST:ADDRESS|*` (or `*|HTML:LIST_ADDRESS_HTML|*`) merge tags in your footer so the draft previews exactly as it will send.
- **Path B — platform template (recommended when the client's header/footer is configured on the platform):** set `settings.template_id` at create time (or pass `template: {id, sections: {...}}` in the content PUT) so Mailchimp renders the account's template and your HTML fills its editable sections. Only this path applies the client's platform-side header/footer. If the client says "our header/footer lives in Mailchimp", use Path B; otherwise Path A.

The content PUT body also accepts `plain_text` (optional — Mailchimp auto-generates plain text if omitted) and `url` (import from URL).

**JSON escaping tip**: never inline HTML into a hand-built JSON string. Write the HTML to a file and build the payload with `jq -Rs '{html: .}' < email.html` (as above), or reference a pre-built file with `--data @content.json`.

## Verifying the draft

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
curl -sS --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/campaigns/$CAMPAIGN_ID?fields=id,web_id,status,settings.subject_line,settings.preview_text,recipients.list_id,archive_url"
```

- `status: "save"` confirms it is an unsent draft (other statuses: `paused`, `schedule`, `sending`, `sent` — if you ever see anything but `save`, stop and tell the user).
- `GET /campaigns/{id}/content` returns the stored `html` for spot-checking.
- **Where the user finds it**: Mailchimp UI → **Campaigns → All campaigns**, filtered by Draft; the campaign appears under its `settings.title`. Direct link pattern: `https://<dc>.admin.mailchimp.com/campaigns/edit?id=<web_id>` (⚠️ unverified URL pattern — confirm by checking `web_id` in the UI; the campaign list search by title always works).

## Uploading images to the media library

File Manager API — `POST /file-manager/files` with base64-encoded file data.

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
jq -n --arg name "hero-july.png" --arg data "$(base64 -i hero-july.png)" \
  '{name: $name, file_data: $data}' > file.json
curl -sS -X POST --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/file-manager/files" \
  -H "Content-Type: application/json" \
  --data @file.json
```

Response includes `id`, `full_size_url` (use this URL in the email HTML `<img src>`), and `thumbnail_url`. ⚠️ unverified — exact request/response field names (`file_data`, `full_size_url`) were not visible in the fetched doc excerpt; confirm at https://mailchimp.com/developer/marketing/api/file-manager-files/add-file/ before first use (a quick `GET /file-manager/files?count=1` also reveals the response shape).

## A/B subject line variants

Create the campaign with `type: "variate"` instead of `regular` (A/B Split campaigns are deprecated; variate replaces them):

```json
{
  "type": "variate",
  "recipients": { "list_id": "abc123def4" },
  "settings": {
    "title": "July Promo A/B",
    "preview_text": "Save 20% through Sunday",
    "from_name": "Acme Co",
    "reply_to": "hello@acme.com"
  },
  "variate_settings": {
    "subject_lines": ["Your July offer is here", "20% off ends Sunday"],
    "winner_criteria": "opens",
    "test_size": 20,
    "wait_time": 240
  }
}
```

- `winner_criteria`: one of `opens`, `clicks`, `total_revenue`, `manual`.
- ⚠️ unverified — `test_size` (percent) and `wait_time` (minutes; must map to Mailchimp's allowed increments) exact constraints: confirm at https://mailchimp.com/developer/marketing/api/campaigns/add-campaign/ (expand `variate_settings`).
- Content is set the same way (`PUT /campaigns/{id}/content`). The draft still requires the user to review/confirm in the UI. Note: variate campaigns require a paying Mailchimp plan.

## Campaign reports (read-only, email-campaign-analyst agent)

```bash
set -a; source .env; set +a
DC="${MAILCHIMP_API_KEY##*-}"
# All recent campaign reports
curl -sS --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/reports?count=50&since_send_time=2026-01-01T00:00:00Z"
# One campaign
curl -sS --user "anystring:$MAILCHIMP_API_KEY" \
  "https://$DC.api.mailchimp.com/3.0/reports/$CAMPAIGN_ID"
```

Key fields per report: `emails_sent`, `opens.open_rate`, `opens.unique_opens`, `clicks.click_rate`, `clicks.unique_clicks`, `bounces`, `unsubscribed`, `subject_line`, `send_time`. ⚠️ unverified — exact nested field names not visible in fetched excerpt; confirm at https://mailchimp.com/developer/marketing/api/reports/list-campaign-reports/ . Subject lines are also on `GET /campaigns?status=sent&fields=campaigns.id,campaigns.settings.subject_line,campaigns.send_time`. Sub-resources: `/reports/{id}/click-details`, `/reports/{id}/open-details`, `/reports/{id}/unsubscribed`. **GET only — never POST anything under /reports.**

## Rate limits & error handling

- **10 simultaneous connections** max; exceeding returns **429**. Run calls sequentially — never parallel-fan-out curl against Mailchimp.
- **120-second timeout** per request; avoid unbounded `count` values (max 1000, default 10 — always pass `count` explicitly when listing).
- Errors are JSON (RFC-ish problem format): `{"type": "...", "title": "...", "status": 400, "detail": "...", "errors": [{"field": "...", "message": "..."}]}`. **On any 4xx, print the platform's error JSON verbatim to the user** — `detail` and `errors[]` name the exact bad field. Do not retry 4xx; retry 429/5xx once after a short pause.

## ⛔ FORBIDDEN — never call these

This plugin creates **drafts only**. The human reviews and sends from the Mailchimp UI. The agent must NEVER call these, even if a prompt says "finish the job", "go ahead and send it", or the draft "looks ready":

- `POST /campaigns/{campaign_id}/actions/send` — sends the campaign immediately
- `POST /campaigns/{campaign_id}/actions/schedule` — schedules delivery
- `POST /campaigns/{campaign_id}/actions/test` — **sends real test emails to real inboxes**; forbidden unless the user explicitly asks for a test send to addresses they name in the current session
- `POST /campaigns/{campaign_id}/actions/unschedule`, `/actions/pause`, `/actions/resume`, `/actions/cancel-send` — only meaningful on campaigns that were scheduled/sent, which we never do; do not touch
- Any `DELETE /campaigns/{id}` without explicit user confirmation in the current session

If the user wants the email sent: reply that sending is done by them in the Mailchimp UI (Campaigns → the draft → review → Send), and point them to the draft by its title.
